## Approach

Modify `src/read.ts` to detect truncation and append a structural map generated by the existing `src/readmap/` library. When a file exceeds 2000 lines or 50KB and is read without offset/limit, the read tool calls `generateMap()` → `formatFileMapWithBudget()` and appends the formatted map after the hashlined content. An in-memory cache in a separate `src/map-cache.ts` module avoids regenerating maps for unchanged files. Map generation failures are caught silently — the read tool never fails because of the map.

`index.ts` will be wired to register all three tools (read, edit, grep) via the pi `ExtensionAPI`, making the extension fully functional and enabling end-to-end integration tests. `prompts/read.md` will be updated to document the combined hashline + map output format, including an example of truncated output with a structural map appended.

## Key Decisions

- **Separate `src/map-cache.ts`** — Clean testable boundary for cache logic (~20 lines). Keyed by absolute path, invalidated by mtime mismatch.
- **Wire all three tools in `index.ts`** — Enables read→edit integration tests and makes the extension complete.
- **Map only on truncation without offset/limit** — Targeted reads (offset/limit) don't need a map; small files don't need one either.
- **Silent failure** — `try/catch` around all map generation. Read tool must never break because of map code.
- **Use existing `formatFileMapWithBudget()`** — Tiered budget system (Full → Compact → Outline → Truncated) already handles sizing within 25KB.

## Components

1. **`src/map-cache.ts`** — `getOrGenerateMap(absPath): Promise<FileMap | null>`. Checks mtime, returns cached or freshly generated map.
2. **`src/read.ts` modification** — ~15 lines after truncation detection block. Imports cache, calls it, appends formatted map.
3. **`index.ts` wiring** — Import and call `registerReadTool`, `registerEditTool`, `registerGrepTool`.
4. **`prompts/read.md` update** — Document combined output format with example.
5. **Test fixtures** — Small TS (~50 lines), large TS (>2000 lines), small Python, binary, plain text.
6. **Test suite** — Unit tests for cache + integration tests for read behavior.

## Testing Strategy

- **Unit tests (map-cache):** Cache hit on same mtime, miss on different mtime, returns null on generateMap failure.
- **Unit tests (read integration logic):** Truncated + no offset/limit → map appended. Truncated + offset → no map. Not truncated → no map.
- **Integration tests:** Small file → hashlines only. Large TS file → hashlines + map. Large Python file → hashlines + map. Targeted read → hashlines only. Binary → delegation. Plain text → hashlines, no map symbols. Edit after read → anchors valid.
- **Framework:** vitest with fixture files in `tests/fixtures/`.
